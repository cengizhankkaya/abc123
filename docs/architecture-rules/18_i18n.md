# Internationalization (i18n) Rules

## Core Principles

- Use feature-first localization structure
- All user-facing strings must be localized
- Use ARB (Application Resource Bundle) format
- Leverage Flutter's `gen_l10n` for code generation
- Support locale switching at runtime
- Provide context for translators

## Project Structure

### Feature-First Localization

**Actual structure:**

Each feature has its own `l10n` directory for localization files:

```text
lib/
├── core/
│   └── l10n/
│       ├── arb/

│       │   ├── app_es.arb            # App-level strings (Spanish)
│       │   └── app_localizations.dart  # Generated
│       └── l10n_extensions.dart      # App context extension (appL10n)
```

**Benefits of this structure:**

- ✅ Localization files live with their features
- ✅ Easy to find and maintain feature-specific strings
- ✅ Features are self-contained (domain, presentation, and l10n together)
- ✅ Easier to delete/move features (everything is in one place)

## Locale Management

### LocaleCubit

**Actual implementation:**

```dart
// lib/core/presentation/bloc/locale_cubit.dart
@immutable
class AppLocale {
  const AppLocale(this.languageCode, [this.countryCode]);

  final String languageCode;
  final String? countryCode;

  /// English locale
  static const en = AppLocale('en');

  /// Spanish locale
  static const es = AppLocale('es');

  static AppLocale fromString(String value) {
    return switch (value) {
      'en' => en,
      'es' => es,
      _ => en,
    };
  }
}

@lazySingleton
class LocaleCubit extends HydratedCubit<AppLocale> {
  LocaleCubit(super.state);

  /// Set locale/language
  void setLocale(AppLocale locale) => emit(locale);

  /// Switch to English
  void setEnglish() => emit(AppLocale.en);

  /// Switch to Spanish
  void setSpanish() => emit(AppLocale.es);

  @override
  AppLocale? fromJson(Map<String, dynamic> json) {
    final languageCode = json['languageCode'] as String?;
    if (languageCode == null) return null;

    final countryCode = json['countryCode'] as String?;
    return AppLocale(languageCode, countryCode);
  }

  @override
  Map<String, dynamic>? toJson(AppLocale state) {
    return {
      'languageCode': state.languageCode,
      'countryCode': state.countryCode,
    };
  }
}
```

## ARB Files

### App-Level Strings

**Example from app_en.arb:**

```json
{
    "@@locale": "en",
    "appName": "Starter App",
    "unexpectedError": "An unexpected error occurred",
    "@unexpectedError": {
        "description": "Error message shown when an unexpected error occurs"
    },
    "serverError": "Server error occurred ({statusCode})",
    "@serverError": {
        "description": "Error message shown when a server/API error occurs",
        "placeholders": {
            "statusCode": {
                "type": "int?"
            }
        }
    },
    "networkError": "Couldn't connect to the server. Please check your connection",
    "@networkError": {
        "description": "Error message shown when there's no network connectivity"
    }
}
```

### Feature-Specific Strings

**Example from auth_en.arb:**

```json
{
    "@@locale": "en",
    "unauthorized": "Invalid credentials or session expired",
    "@unauthorized": {
        "description": "Error message shown when user authentication fails"
    },
    "emailLabel": "Email",
    "@emailLabel": {
        "description": "Label for email input field"
    },
    "welcomeBack": "Welcome back, {email}!",
    "@welcomeBack": {
        "description": "Welcome back message for existing users",
        "placeholders": {
            "email": {
                "type": "String"
            }
        }
    },
    "login": "Login",
    "@login": {
        "description": "Button text to login"
    }
}
```

### ARB File Rules

- ✅ **DO**: Use `@@locale` at the top of each file
- ✅ **DO**: Provide `@key` descriptions for translators
- ✅ **DO**: Define placeholders with types
- ✅ **DO**: Use descriptive keys (not generic like `label1`)
- ✅ **DO**: Keep feature strings in feature-specific files
- ❌ **DON'T**: Hardcode strings in code
- ❌ **DON'T**: Use keys without descriptions
- ❌ **DON'T**: Mix features in the same ARB file

## Context Extensions

### Accessing Localizations

**Actual implementation:**

Each feature has its own `l10n_extensions.dart` file with a context extension for accessing its localizations:

```dart
// lib/core/l10n/l10n_extensions.dart
extension AppLocalizationsX on BuildContext {
  /// Get app-level localizations
  AppLocalizations get appL10n => AppLocalizations.of(this);
}
```

```dart
// lib/features/auth/l10n/l10n_extensions.dart
extension AuthLocalizationsX on BuildContext {
  /// Get auth feature localizations
  AuthLocalizations get authL10n => AuthLocalizations.of(this);
}
```

```dart
// lib/features/home/l10n/l10n_extensions.dart
extension HomeLocalizationsX on BuildContext {
  /// Get home feature localizations
  HomeLocalizations get homeL10n => HomeLocalizations.of(this);
}
```

```dart
// lib/features/profile/l10n/l10n_extensions.dart
extension ProfileLocalizationsX on BuildContext {
  /// Get profile feature localizations
  ProfileLocalizations get profileL10n => ProfileLocalizations.of(this);
}
```

```dart
// lib/features/settings/l10n/l10n_extensions.dart
extension SettingsLocalizationsX on BuildContext {
  /// Get settings feature localizations
  SettingsLocalizations get settingsL10n => SettingsLocalizations.of(this);

```

**Benefits of this structure:**

- ✅ Each feature exports only its own localization extension
- ✅ Clear feature boundaries - no cross-feature dependencies
- ✅ Import only what you need (better for code splitting)
- ✅ Easier to understand which localizations a file uses

## Usage in Widgets

### Basic Usage

```dart
// ✅ Import feature-specific l10n extension
import 'package:starter_app/features/auth/l10n/l10n_extensions.dart';

class LoginPage extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(
        // ✅ Use feature-specific l10n extension
        title: Text(context.authL10n.login),
      ),
      body: Column(
        children: [
          // ✅ Access strings via context extension
          TextField(
            decoration: InputDecoration(
              labelText: context.authL10n.emailLabel,
              hintText: context.authL10n.emailHint,
            ),
          ),
          ElevatedButton(
            onPressed: _handleLogin,
            child: Text(context.authL10n.login),
          ),
        ],
      ),
    );
  }
}
```

**Import Guidelines:**

- ✅ Import `lib/features/{feature}/l10n/l10n_extensions.dart` when using feature-specific strings
- ✅ Import `lib/core/l10n/l10n_extensions.dart` when using app-level strings
- ✅ Import `lib/core/presentation/bloc/bloc.dart` when using LocaleCubit
- ✅ Can import multiple l10n extension files if the widget uses strings from multiple features

### In BLoCs (Avoid)

```dart
// ❌ DON'T access l10n in BLoCs (no BuildContext)
class AuthBloc extends Bloc<AuthEvent, AuthState> {
  void _onError() {
    // ❌ Can't do this - no context
    final message = context.authL10n.unauthorized;
  }
}

// ✅ DO: Return domain failures, map to l10n in UI
class AuthBloc extends Bloc<AuthEvent, AuthState> {
  void _onError() {
    // ✅ Return domain failure
    emit(state.copyWith(
      error: ErrorModel.fromFailure(AuthFailure.unauthorized()),
    ));
  }
}

// ✅ Map to localized strings in UI layer
BlocListener<AuthBloc, AuthState>(
  listener: (context, state) {
    final error = state.error;
    if (error != null) {
      final message = error.getMessage(context, failureMessageService);
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text(message)),
      );
    }
  },
)
```

## Configuration

### No l10n.yaml Required

This project does **NOT** use `l10n.yaml`. Instead, we use a shell script to generate localizations for each feature with custom parameters.

### pubspec.yaml

```yaml
flutter:
  # Note: generate: true is NOT used
```

## Code Generation

### Using generate_localizations.sh

**Actual implementation:**

```bash
# generate_localizations.sh
#!/bin/bash

echo "Generating Auth localizations..."
flutter gen-l10n --arb-dir ./lib/features/auth/l10n --template-arb-file auth_en.arb --output-localization-file auth_localizations.dart --output-class AuthLocalizations --no-nullable-getter

echo "Generating Home localizations..."
flutter gen-l10n --arb-dir ./lib/features/home/l10n --template-arb-file home_en.arb --output-localization-file home_localizations.dart --output-class HomeLocalizations --no-nullable-getter

echo "Generating Profile localizations..."
flutter gen-l10n --arb-dir ./lib/features/profile/l10n --template-arb-file profile_en.arb --output-localization-file profile_localizations.dart --output-class ProfileLocalizations --no-nullable-getter

echo "Generating Settings localizations..."
flutter gen-l10n --arb-dir ./lib/features/settings/l10n --template-arb-file settings_en.arb --output-localization-file settings_localizations.dart --output-class SettingsLocalizations --no-nullable-getter

echo "Generating App localizations..."
flutter gen-l10n --arb-dir ./lib/core/l10n/arb --template-arb-file app_en.arb --output-localization-file app_localizations.dart --output-class AppLocalizations --no-nullable-getter

echo "Localization generation completed!"
```

### Running the Script

```bash
# Make script executable (first time only)
chmod +x generate_localizations.sh

# Generate all localizations
./generate_localizations.sh
```

**Why use a script instead of l10n.yaml?**

- ✅ Allows feature-specific ARB directories (not supported by l10n.yaml)
- ✅ Custom output class names per feature
- ✅ Explicit control over generation parameters
- ✅ Easier to customize per-feature settings

## Locale Switching

### In Settings

```dart
class SettingsPage extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return BlocBuilder<LocaleCubit, AppLocale>(
      builder: (context, locale) {
        return Column(
          children: [
            ListTile(
              title: Text(context.settingsL10n.language),
              subtitle: Text(_getLanguageName(locale)),
              onTap: () => _showLanguagePicker(context),
            ),
          ],
        );
      },
    );
  }

  String _getLanguageName(AppLocale locale) {
    return switch (locale.languageCode) {
      'en' => 'English',
      'es' => 'Español',
      _ => 'English',
    };
  }

  void _showLanguagePicker(BuildContext context) {
    showDialog(
      context: context,
      builder: (context) => AlertDialog(
        title: Text(context.settingsL10n.selectLanguage),
        content: Column(
          mainAxisSize: MainAxisSize.min,
          children: [
            ListTile(
              title: const Text('English'),
              onTap: () {
                context.read<LocaleCubit>().setEnglish();
                Navigator.pop(context);
              },
            ),
            ListTile(
              title: const Text('Español'),
              onTap: () {
                context.read<LocaleCubit>().setSpanish();
                Navigator.pop(context);
              },
            ),
          ],
        ),
      ),
    );
  }
}
```

## Best Practices

### String Keys

```dart
// ✅ GOOD - Descriptive keys
"emailLabel": "Email"
"emailHint": "Enter your email address"
"loginButton": "Login"
"welcomeBackMessage": "Welcome back, {name}!"

// ❌ BAD - Generic keys
"label1": "Email"
"hint1": "Enter your email address"
"button1": "Login"
"message1": "Welcome back, {name}!"
```

### Descriptions

```dart
// ✅ GOOD - Clear context for translators
"@emailLabel": {
    "description": "Label for email input field in login form"
}

// ❌ BAD - No context
"@emailLabel": {
    "description": "Email"
}

// ❌ BAD - No description at all
"emailLabel": "Email"
```

### Placeholders

```dart
// ✅ GOOD - With type
"welcomeBack": "Welcome back, {email}!",
"@welcomeBack": {
    "description": "Welcome message for returning users",
    "placeholders": {
        "email": {
            "type": "String"
        }
    }
}

// ✅ GOOD - Multiple placeholders
"orderSummary": "You ordered {count} items for ${total}",
"@orderSummary": {
    "placeholders": {
        "count": {"type": "int"},
        "total": {"type": "double"}
    }
}
```

## Localization Checklist

### Pre-Release i18n Review

- [ ] All user-facing strings localized
- [ ] No hardcoded strings in UI
- [ ] All ARB files have descriptions
- [ ] Placeholders properly typed
- [ ] Locale switching works
- [ ] All supported locales tested
- [ ] RTL languages tested (if supported)
- [ ] Date/number formatting locale-aware
- [ ] No UI breaks with long translations

### Common i18n Mistakes

- ❌ Hardcoding strings in code
- ❌ Using l10n in BLoCs
- ❌ Missing descriptions in ARB files
- ❌ Generic key names
- ❌ Not testing locale switching
- ❌ Forgetting to generate after ARB changes
- ❌ Mixing features in one ARB file

## Adding New Languages

### Steps to Add a Language

1. Create new ARB files for all features:

    ```text
    lib/core/l10n/arb/app_fr.arb
    lib/features/auth/l10n/auth_fr.arb
    lib/features/home/l10n/home_fr.arb
    lib/features/profile/l10n/profile_fr.arb
    lib/features/settings/l10n/settings_fr.arb
    ```

2. Add locale to AppLocale class:

    ```dart
    // lib/core/presentation/bloc/locale_cubit.dart
    class AppLocale {
      static const en = AppLocale('en');
      static const es = AppLocale('es');
      static const fr = AppLocale('fr');  // ✅ Add new locale

      static AppLocale fromString(String value) {
        return switch (value) {
          'en' => en,
          'es' => es,
          'fr' => fr,  // ✅ Add to switch
          _ => en,
        };
      }
    }
    ```

3. Add convenience method to LocaleCubit:

    ```dart
    // lib/core/presentation/bloc/locale_cubit.dart
    class LocaleCubit extends HydratedCubit<AppLocale> {
      void setFrench() => emit(AppLocale.fr);  // ✅ Add setter method
    }
    ```

4. Add to supported locales in MaterialApp:

    ```dart
    supportedLocales: const [
      Locale('en'),
      Locale('es'),
      Locale('fr'),  // ✅ Add to supported
    ],
    ```

5. Generate localization code:

    ```bash
    # Run the generation script
    ./generate_localizations.sh
    ```

**Note:** The script already handles all features, so you don't need to modify it. Just ensure your new ARB files follow the naming convention (`{feature}_{languageCode}.arb`).
